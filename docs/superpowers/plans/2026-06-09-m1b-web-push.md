# M1b — Phone Push (Web Push / PWA) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development or executing-plans. Steps use `- [ ]`.

**Goal:** Get a push notification on the user's phone when a decision needs them; tapping Allow/Deny on the notification resolves it and Claude continues — without opening the app.

**Architecture:** Web Push (VAPID) + a PWA service worker. No FCM/APNs — the browser's push service delivers, the API holds VAPID keys and signs pushes with the `web-push` library. Hexagonal: domain `PushSubscriptionStore` + `PushSender` ports; pg/in-memory + web-push/noop adapters. A bus listener fans a push out on `decision.created` for attention-worthy kinds. Service worker fetches the existing `/api/proxy/decisions/:id/resolve` (cookie auth) for Allow/Deny actions.

**Tech Stack:** NestJS, `web-push`, Next.js service worker + PushManager, Postgres migration, Zod shared schema.

## Task 1: shared PushSubscription schema
**Files:** Create `packages/shared/src/push/push.ts`; Modify `packages/shared/src/index.ts`.
- `PushSubscriptionSchema` = `{ endpoint: string.url, keys: { p256dh: string, auth: string }, label?: string }` (the standard browser subscription JSON + optional device label).
- Export from barrel. Build shared.

## Task 2: migration
**Files:** Create `apps/api/migrations/005_push_subscriptions.sql`.
- `push_subscriptions(id uuid pk, endpoint text unique not null, p256dh text not null, auth text not null, label text, created_at timestamptz default now(), last_used_at timestamptz)`. Idempotent (`create table if not exists`).

## Task 3: domain ports
**Files:** Create `apps/api/src/domain/push/push-subscription-store.port.ts`, `apps/api/src/domain/push/push-sender.port.ts`.
- `PushSubscriptionStore`: `save(sub)`, `list()`, `removeByEndpoint(endpoint)`. Symbol token.
- `PushSender`: `send(sub, payload: string): Promise<"ok" | "gone" | "error">`. Symbol token. `"gone"` = 404/410 (prune).

## Task 4: persistence adapters
**Files:** Create `apps/api/src/adapters/persistence/in-memory-push-subscription-store.ts`, `postgres-push-subscription-store.ts`. Tests for in-memory.
- save = upsert on endpoint. list = all. removeByEndpoint = delete.

## Task 5: push sender adapters
**Files:** Create `apps/api/src/adapters/push/web-push-sender.ts`, `noop-push-sender.ts`. Add `web-push` dep to `apps/api/package.json`.
- WebPushSender: ctor sets VAPID details from `VAPID_PUBLIC_KEY/PRIVATE_KEY/SUBJECT`; `send` → `webpush.sendNotification`; map statusCode 404/410 → `"gone"`, others throw → `"error"`.
- NoopPushSender: returns `"ok"`, used when VAPID keys absent (and in tests).

## Task 6: PushService (application) + bus fan-out
**Files:** Create `apps/api/src/application/push.service.ts`. Test.
- `register(sub)`, `list()`, `remove(endpoint)`, `vapidPublicKey()`.
- `notify(decision)`: build payload `{title, body, url:"/", decisionId, kind, deliverable, options}`; for each subscription `sender.send`; prune `"gone"`.
- `onModuleInit`: subscribe to `EventsBus.stream$`; on `decision.created` with kind in {question, permission, notification}, call `notify`.

## Task 7: HTTP controller
**Files:** Create `apps/api/src/adapters/http/push.controller.ts`. e2e test.
- `GET /push/vapid` → `{ publicKey }`.
- `POST /push/subscriptions` → register (Zod parse).
- `POST /push/subscriptions/remove` → `{ endpoint }` remove (DELETE-with-body is awkward through the proxy; use POST).
- Guarded by `InstanceTokenGuard`.

## Task 8: wire module + env
**Files:** Modify `apps/api/src/app.module.ts`, `.env.example`, `docker-compose.yml`, `deploy/remote.sh`/server `.env`.
- Providers/factories for the two stores + sender (web-push when keys present, else noop), PushService; register controller.
- Generate VAPID keys; add `VAPID_PUBLIC_KEY/PRIVATE_KEY/SUBJECT` to `.env.example` (placeholders) and the server `.env`; pass through to api service in compose.

## Task 9: web — manifest + service worker
**Files:** Create `apps/web/public/manifest.webmanifest`, `apps/web/public/sw.js`. Modify `apps/web/app/layout.tsx` (manifest link).
- manifest: name, short_name, start_url "/", display "standalone", background/theme color, icons.
- sw.js: `push` → `showNotification(title, {body, data, actions:[Allow,Deny] when kind==="permission"})`; `notificationclick` → action allow/deny → `fetch('/api/proxy/decisions/<id>/resolve', {method:POST, credentials:'include', body:{choice}})`; else `clients.openWindow(data.url)`.

## Task 10: web — subscribe UI + SW registration
**Files:** Create `apps/web/components/NotificationsToggle.tsx`; Modify `apps/web/app/page.tsx`, `apps/web/app/api/proxy/[...path]/route.ts`.
- Register `/sw.js` on load. Toggle: request permission → GET `/api/proxy/push/vapid` → `pushManager.subscribe({userVisibleOnly:true, applicationServerKey})` → POST `/api/proxy/push/subscriptions`. Show enabled/blocked/enable state; place at top of Situation Room.
- Proxy ALLOWED += `/^push\/vapid$/`, `/^push\/subscriptions$/`, `/^push\/subscriptions\/remove$/`.

## Task 11: deploy + live phone test
- Migrate, deploy, smoke. Generate + set VAPID on server. User enables notifications, triggers a permission decision, gets a phone push, taps Allow → Claude continues.
